# Module 1: 数据底座 — 数据源分析

> 本文档是 Module 1 正式设计（接口定义 + 数据模型）的前置输入。
> 确定"从哪里取数据、取什么、怎么缓存、怎么去重、怎么排序"后，再进入接口设计。

## 1. 职责边界

**做：**
- 统一获取所有外部金融数据（行情/估值/财务/板块/新闻）
- 本地持久化缓存（SQLite）+ 内存缓存
- 数据清洗、去重、标准化
- 提供标准化数据 API 供下游模块消费
- 按历史深度初始化数据（指数 20 年 / 板块 15 年 / 个股 10 年）

**不做：**
- 估值计算（归 Module 4）
- 新闻加工处理（归 Module 2）
- 用户数据管理（归 Module 0）
- 直接对外展示（归 Module 3/5/6）

**核心约束：** 数据获取唯一入口 — 其他模块不直接调用外部 API。

## 3. 缓存策略

### 3.1 缓存分层架构

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 内存缓存（进程内 dict / TTL 缓存）      │
│  - 实时行情（TTL: 60s）                           │
│  - 板块实时排行（TTL: 60s）                       │
│  - 热门数据（高频访问的个股/指数信息）              │
└──────────────────────┬──────────────────────────┘
                       │ 缓存 miss / TTL 过期
                       ▼
┌─────────────────────────────────────────────────┐
│  Layer 2: SQLite 持久化缓存                       │
│  - 历史 K 线（永久存储）                           │
│  - 板块列表/成分股（TTL: 24h）                     │
│  - 估值指标（TTL: 24h）                           │
│  - 财务报表（TTL: 7d）                            │
│  - 新闻（TTL: 72h）                               │
└──────────────────────┬──────────────────────────┘
                       │ 缓存 miss / TTL 过期
                       ▼
┌─────────────────────────────────────────────────┐
│  Layer 3: 外部数据源调用                          │
│  - AKShare → 东方财富/同花顺/乐股/中证            │
│  - 腾讯财经 API                                   │
│  - 金十数据 API                                   │
└─────────────────────────────────────────────────┘
```

### 3.2 各数据类型缓存规则

| 数据类型 | 缓存层 | TTL | 存储位置 | 刷新策略 | 说明 |
|---------|--------|-----|---------|---------|------|
| 实时行情（个股/ETF） | L1 + L2 | L1: 60s / L2: 永久（当日快照） | 内存 + SQLite | 交易时段每分钟轮询 | 非交易时段读 L2 最后快照 |
| 指数实时行情 | L1 + L2 | L1: 60s / L2: 永久 | 内存 + SQLite | 交易时段每分钟轮询 | 全球指数需注意时区差异 |
| 板块实时行情 | L1 + L2 | L1: 60s / L2: 永久 | 内存 + SQLite | 交易时段每分钟轮询 | — |
| 历史 K 线 | L2 | 永久 | SQLite | 每日 16:30 增量追加 | 已入库的数据不重复拉取 |
| 估值指标 | L2 | 24h | SQLite | 每日 18:00 批量更新 | 交易日才更新 |
| 财务报表 | L2 | 7d | SQLite | 每周检查一次 | 季报期可缩短到 1d |
| 板块列表 | L2 | 24h | SQLite | 每日 18:00 全量刷新 | — |
| 成分股 | L2 | 24h | SQLite | 每日 18:00 全量刷新 | 指数成分股按季度 |
| 资金流向 | L1 + L2 | L1: 5min / L2: 永久 | 内存 + SQLite | 交易时段每 5 分钟 | — |
| 基金净值 | L2 | 24h | SQLite | 每日晚间更新 | — |
| 新闻 | L2 | 72h | SQLite | 每 30 分钟拉取增量 | 超 72h 自动清理 |
| 融资融券汇总 | L2 | 永久 | SQLite | 每日 18:00 更新当日数据 | 历史数据永久存储，用于百分位计算 |

### 3.3 缓存键设计

| 数据类型 | 缓存键格式 | 示例 |
|---------|-----------|------|
| 个股实时 | `quote:{market}:{symbol}` | `quote:A:600519` |
| 指数实时 | `index:{market}:{symbol}` | `index:A:000300` |
| 板块实时 | `board:{type}:{code}` | `board:industry:BK0477` |
| 资金流向 | `flow:{type}:{date}` | `flow:northbound:20260507` |

## 4. 去重策略

### 4.1 去重原则

**入库前必查，按业务键去重。** 不同数据类型有不同的唯一键定义。

### 4.2 各数据类型去重规则

| 数据类型 | 去重键 | 去重方式 | 说明 |
|---------|--------|---------|------|
| 历史 K 线 | `(symbol, date)` | INSERT OR IGNORE | 同一标的同一天只有一条，已入库的不覆盖 |
| 实时行情 | `symbol` | 内存覆盖 | 每次拉取全量覆盖，天然去重 |
| 估值指标 | `(symbol, date)` | INSERT OR REPLACE | 当日数据覆盖更新（估值可能修正） |
| 财务报表 | `(symbol, report_type, report_date)` | INSERT OR REPLACE | 同期报表覆盖更新（财报更正） |
| 板块列表 | `sector_code` | INSERT OR REPLACE | 板块代码全局唯一 |
| 成分股 | `(sector_code, stock_code)` | INSERT OR IGNORE | 同一股在同板块不重复 |
| 基金净值 | `(fund_code, nav_date)` | INSERT OR IGNORE | 同基金同日只有一条净值 |
| 新闻 | `(source_id)` 或 `(title_hash, publish_time)` | INSERT OR IGNORE | 金十数据自带 news_id 作为去重键 |
| 资金流向 | `(flow_type, date)` | INSERT OR REPLACE | 同类型同日覆盖 |
| 融资融券汇总 | `(date)` | INSERT OR REPLACE | 沪深合计，每日一条；当日可能修正 |

### 4.3 去重实现策略

```python
# 历史 K 线：增量追加，已存在则跳过
INSERT OR IGNORE INTO stock_daily_kline (symbol, date, open, high, low, close, volume)
VALUES (?, ?, ?, ?, ?, ?, ?);

# 估值指标：当日覆盖（可能有修正值）
INSERT OR REPLACE INTO stock_valuation (symbol, date, pe_ttm, pb, ps_ttm, market_cap)
VALUES (?, ?, ?, ?, ?, ?);

# 新闻：按 source_id 去重
INSERT OR IGNORE INTO news_items (source, source_id, title, content, publish_time)
VALUES (?, ?, ?, ?, ?);
```

## 5. 排序策略

### 5.1 数据库层排序

数据库查询时的默认 ORDER BY 规则：

| 数据类型 | 默认排序 | 说明 |
|---------|---------|------|
| 历史 K 线 | `date ASC` | 按时间正序，便于图表渲染和趋势计算 |
| 实时行情列表 | 按用户自定义 `sort_order` | 跟随 Module 0 用户自选股顺序 |
| 板块实时排行 | `change_pct DESC` | 按涨跌幅降序，热门板块优先 |
| 成分股列表 | `weight DESC` / `market_cap DESC` | 按权重或市值降序 |
| 新闻列表 | `publish_time DESC` | 最新新闻优先 |
| 资金流向 | `net_inflow DESC` | 净流入金额降序 |
| 基金净值 | `nav_date DESC` | 最新净值优先 |
| 估值指标 | `date DESC` | 最新估值优先 |
| 融资融券汇总 | `date DESC` | 最新数据优先，百分位计算时切换为 `date ASC` |

### 5.2 业务层排序

部分场景需要 Service 层二次排序：

| 场景 | 排序依据 | 说明 |
|------|---------|------|
| 估值百分位分组 | `percentile ASC` → 分高估/适中/低估三组 | Module 4 计算后排序 |
| 市场温度 | 综合涨跌比 + 资金流向 + 波动率 | Module 3 计算后排序 |
| 板块热度 | 综合涨跌幅 + 资金流入 + 成交量变化 | Module 5 计算后排序 |

## 6. 数据初始化策略

### 6.1 设计原则

**大盘指数主动拉取，板块/个股按需触发。**

- 首次部署只拉取三大市场（A/港/美）的指定大盘指数，作为系统基础数据
- 行业/概念板块、个股、ETF 等数据，等用户添加到自选或关注后再触发拉取
- 不默认拉取全量板块、概念及个股数据，避免无意义的数据堆积和外部源压力

### 6.2 首次部署初始化清单

系统首次部署后，自动拉取以下 **10 个核心指数** 的 20 年日 K 数据：

#### A 股市场

| 科创 50 | `000688` | 科技成长代表 |

#### 港股市场

首次部署后，同时自动拉取以下市场级数据：

| 核心指数估值历史（PE/PB） | **10 年** | ~2500 条/指数 × 10 ≈ 2.5 万条 | Module 4 估值百分位计算必需 |

> **注：** 融资融券和指数估值数据与核心指数 K 线同时初始化，因为它们是 Module 4 的基础输入，且数据量小、拉取快。

### 6.3 按需触发初始化（用户行为驱动）

| 触发事件 | 拉取内容 | 历史深度 | 说明 |
|---------|---------|---------|------|
| 用户添加个股到自选 | 该个股日 K + 估值指标 + 财务摘要 | **10 年** | 异步任务，后台执行 |
| 用户添加 ETF/基金到自选 | 该基金日 K + 净值 | **10 年** | 异步任务 |
| 用户添加行业板块关注 | 该板块日 K + 成分股 | **15 年** | 异步任务 |
| 用户添加概念板块关注 | 该概念日 K + 成分股 | **15 年** | 异步任务 |
| 用户添加港股个股到自选 | 该港股日 K + 估值指标 | **10 年** | 异步任务 |

> **关键：** 拉取任务在后台异步执行，前端显示"数据加载中"状态，不阻塞用户操作。

### 6.4 初始化流程

```
首次部署
    │
    ▼
拉取 10 个核心指数 20 年日 K（P0，自动执行）
    │
    ▼
拉取核心指数的估值历史 + 融资融券 10 年历史（P0，自动执行）
    │
    ▼
系统就绪，等待用户注册
    │
    ▼
[用户注册] → 创建默认自选列表（空）
    │
    ▼
[用户添加个股/ETF 到自选] → 后台拉取该标的 10 年日 K
    │
    ▼
[用户添加板块/概念关注] → 后台拉取该板块 15 年日 K + 成分股
```

|------|------|------|
| 核心指数日 K 增量更新 | 每日 16:30 | 追加当日新 K 线 |
| 用户自选股日 K 增量更新 | 每日 16:30 | 追加所有已添加标的当日新 K 线 |
| 用户关注板块日 K 增量更新 | 每日 16:30 | 追加所有已关注板块当日新 K 线 |
| 估值指标更新 | 每日 18:00 | 更新所有已入库标的的估值数据 |
| 板块成分股刷新 | 每日 18:00 | 刷新所有已关注板块的成分股 |
| 新闻拉取 | 每 30 分钟 | 拉取金十快讯增量 |
| 融资融券数据更新 | 每日 18:00 | 拉取沪深合计当日融资融券数据 |
| 财务报表检查 | 每周一次 | 检查已入库个股是否有新财报 |

### 6.6 限流策略

AKShare 底层爬取公开网站，需控制调用频率避免被封。

| 策略 | 参数 | 说明 |
|------|------|------|
| 单接口调用间隔 | ≥ 0.5 秒 | 避免触发反爬 |
| 批量拉取间隔 | ≥ 2 秒/标的 | 历史 K 线按标的逐个拉取 |
| 并发限制 | 最多 1 个并发 | AKShare 不支持并发调用 |
| 失败重试 | 最多 3 次，指数退避 | 1s → 3s → 9s |
| 熔断机制 | 连续 5 次失败暂停 5 分钟 | 避免持续触发限流 |

## 9. 技术选型确认

| 选项 | 选择 | 理由 |
|------|------|------|
| 核心数据获取库 | AKShare | 免费开源、覆盖全、Python 原生 |
| 实时行情辅助源 | 腾讯财经 API | 速度快、无鉴权、项目已有约定 |
| 新闻数据源 | 金十数据 | 权威、7×24 快讯、MVP 单一来源 |
| 本地持久化 | SQLite（WAL 模式） | 零运维、与项目架构一致 |
| 内存缓存 | 进程内 dict + TTL | MVP 够用，未来可升级 Redis |
| 定时任务 | APScheduler | Python 原生、轻量、支持 cron 表达式 |
| 港股支持 | MVP 包含 | 港股通存在，数据有实际需求 |
| 美股支持 | MVP 后置 | 功能实现优先，后续增补 |
| 数据粒度 | 日 K 存储，周 K 按需聚合 | 减少存储冗余，保持灵活性 |

## 10. 设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 数据获取唯一入口 | Module 1 统一获取 | system-patterns.md 约束，其他模块不直接调外部 API |
| 大盘指数历史深度 | 20 年 | 看牛熊周期需要足够长的数据 |
| 板块历史深度 | 15 年 | 看行业规律，板块生命周期较长 |
| 个股/基金历史深度 | 10 年 | 很多标的成立不足 10 年，按最长取 |
| 数据粒度 | 日 K 为主 | 非量化交易，日 K 足够；周 K 通过聚合生成 |
| 实时行情频率 | 分钟级 | 操作按天进行，非量化交易，分钟级足够 |
| 新闻数据源 | 金十数据 | MVP 阶段单一来源，权威可靠，未来增补 |
| 港股个股 | MVP 包含 | 港股通存在，覆盖港股个股有实际需求 |
| 美股个股 | MVP 后置 | 功能实现优先，后续根据用户需求增补 |
| 缓存分层 | 内存 + SQLite | 简单够用，无需引入 Redis 等外部依赖 |
| 历史 K 线存储 | 日 K 存储，聚合生成周 K/月 K | 减少存储冗余，灵活性高 |
| 初始化策略 | 大盘指数主动拉取，板块/个股按需触发 | 不默认拉取全量数据，避免无意义堆积；用户添加后异步拉取 |
| 首批指数清单 | 10 个核心指数（A 股 6 + 港股 2 + 美股 2） | 覆盖三大市场基准，约 5 万条数据，5-10 分钟完成 |
| 融资融券纳入 MVP | 沪深合计汇总级别，10 年历史 | 反映市场杠杆水平，融资余额百分位+融资/市值比是估值分析核心热度指标 |
| 融资融券粒度 | MVP 仅汇总级别，个股明细后置 | 个股级数据量大且非核心需求，汇总级别足够计算市场整体杠杆水平 |
| 指数估值历史初始化 | 首次部署同步拉取 10 年历史 | PE/PB 百分位计算需要足够长的历史序列，必须在系统可用前就绪 |
