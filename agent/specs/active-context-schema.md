# Active Context Schema 规范

> 定义 `agent/memory-bank/active-context.md` 的统一结构、更新规则和冲突处理机制。
> 所有角色更新此文件时必须遵循本规范。

---

## 文件定位

`active-context.md` 是**所有角色的共同入口**，是 Agent 切换时的"握手文件"。
它的作用是：让任何角色在任意时刻都能快速了解项目当前状态。

**核心原则：**
- 只写**状态**和**引用**，不写**详细内容**
- 详细内容写在专门的文档中，这里只放链接
- 保持**最新**，过期信息及时清理

---

## 统一章节结构

```markdown
# 当前焦点 (Active Context)

## 当前阶段

[一句话描述当前阶段，如："MVP 编码阶段 — Phase 2/3 已完成，准备进入 M4"]

### 最近完成
- [日期] [事项] [状态]

### 下一步行动
- [ ] [事项] [负责人] [优先级]

## 任务流水线状态

| 需求 | 产品 | 架构 | 编码 | 审查 | 状态 |
|------|------|------|------|------|------|
| {feature} | ✅ | ✅ | 🔄 | ⏳ | 编码中 |

## API 验证状态

- GET /path ✅
- POST /path ❌ [原因]

## 数据状态

| 标的 | K线 | 估值 | 实时行情 |
|------|-----|------|----------|
| {symbol} | {count}条 | {count}条 | ✅/❌ |

## 已知阻塞

- [ ] [问题描述] [影响范围] [预计解决时间]

## 技术债务

- [ ] [债务描述] [优先级] [计划解决时间]

## 最近变更

| 日期 | 文件 | 变更类型 | 变更摘要 | 作者 |
|------|------|----------|----------|------|
| YYYY-MM-DD | path | 新增/修改/删除 | 摘要 | 角色 |
```

## 各角色更新规则

### 产品经理 (Product Manager)

**可更新的章节：**
- `## 当前阶段` — 更新阶段描述
- `## 任务流水线状态` — 更新需求状态（产品列）
- `## 下一步行动` — 添加产品相关的下一步
- `## 已知阻塞` — 添加需求相关的阻塞
- `## 最近变更` — 记录需求文档变更

**更新模板：**
```markdown
## 当前待设计需求
- {feature-name}：{一句话描述}，优先级 P0/P1/P2，需求文档 docs/features/{feature-name}.md
```

**禁止：**
- 不更新 API 验证状态（编码者负责）
- 不更新数据状态（编码者负责）
- 不更新技术债务（架构师负责）

---

### 架构师 (Architect)

**可更新的章节：**
- `## 当前阶段` — 更新阶段描述
- `## 任务流水线状态` — 更新设计状态（架构列）
- `## 下一步行动` — 添加架构相关的下一步
- `## 已知阻塞` — 添加设计相关的阻塞
- `## 技术债务` — 添加架构相关的债务
- `## 最近变更` — 记录设计文档变更

**更新模板：**
```markdown
## 当前待编码模块
- Module {N}：{模块名}，设计文档 docs/modules/module-{N}-{name}.md，依赖：{无/Module X}
```

**禁止：**
- 不更新 API 验证状态（编码者负责）
- 不更新数据状态（编码者负责）

---

### 编码者 (Coder)

**可更新的章节：**
- `## 当前阶段` — 更新阶段描述
- `## 任务流水线状态` — 更新编码状态（编码列）
- `## API 验证状态` — 更新 API 测试结果
- `## 数据状态` — 更新数据同步状态
- `## 下一步行动` — 添加编码相关的下一步
- `## 已知阻塞` — 添加编码相关的阻塞
- `## 最近变更` — 记录代码变更

**更新模板：**
```markdown
## 当前待审查
- Module {N}：{模块名}，PR #{number}，开发日志 agent/memory/YYYY-MM-DD.md
```

**禁止：**
- 不更新技术债务（架构师负责）
- 不修改需求状态（产品经理负责）

---

### 审查者 (Reviewer)

**可更新的章节：**
- `## 当前阶段` — 更新阶段描述
- `## 任务流水线状态` — 更新审查状态（审查列）
- `## 下一步行动` — 添加审查相关的下一步
- `## 最近变更` — 记录验收报告变更

**更新模板（通过）：**
```markdown
## 最近完成
- Module {N}：{模块名}，验收报告 docs/acceptance-report-YYYYMMDD.md
```

**更新模板（不通过）：**
```markdown
## 当前待修复
- Module {N}：{模块名}，PR #{number}，问题见验收报告 docs/acceptance-report-YYYYMMDD.md
```

**禁止：**
- 不更新 API 验证状态（编码者负责）
- 不更新数据状态（编码者负责）
- 不更新技术债务（架构师负责）

## 冲突处理机制

### 场景 1：两个角色同时更新同一章节

**处理原则：** 后更新者覆盖先更新者，但必须在变更日志中说明。

**示例：**
```markdown
## 最近变更
| 日期 | 文件 | 变更类型 | 变更摘要 | 作者 |
|------|------|----------|----------|------|
| 2026-05-13 | active-context.md | 修改 | 更新 API 状态：/api/v1/auth 从 ❌ 改为 ✅ | coder |
| 2026-05-13 | active-context.md | 修改 | 更新阶段：从"编码中"改为"待审查" | reviewer |
```

### 场景 2：角色 A 更新了角色 B 的专属章节

**处理原则：** 允许更新，但必须在变更日志中标注，并通知相关角色。

**示例：**
编码者发现技术债务，更新了 `## 技术债务`：
```markdown
## 技术债务
- [ ] Module 4 估值计算性能优化（当前 O(n²)，可优化至 O(n)）
  - 发现者：coder
  - 计划解决：architect（下次迭代）
```

### 场景 3：对状态标记有争议

**处理原则：** 以**下游角色**的状态为准。

**示例：**
- 产品经理标记"需求已完成"
- 架构师认为"设计未完成"
- **以架构师为准**（下游角色更了解实际情况）

升级路径：
1. 在 `agent/memory/YYYY-MM-DD.md` 中记录争议
2. 涉及角色在日志中各自陈述理由
3. 如无法达成一致，由**产品经理**最终决策（产品对价值负责）

## 更新频率

| 场景 | 更新时机 | 必须更新 |
|------|----------|----------|
| 会话开始时 | 读取当前状态 | 否（只读） |
| 产生产出物时 | 立即更新对应章节 | 是 |
| 发现阻塞时 | 立即更新阻塞章节 | 是 |
| 会话结束时 | 更新最近变更 | 是 |
| 每日 | 检查并清理过期信息 | 建议 |

---

## 清理规则

**自动清理（建议每日执行）：**
- `## 最近完成` 中超过 7 天的项 → 移动到 `agent/memory-bank/progress.md`
- `## 已知阻塞` 中已解决的项 → 标记为 ✅ 并保留 3 天后删除
- `## 最近变更` 中超过 30 天的项 → 删除（历史在 Git 中保留）

**手动清理：**
- 阶段变更时（如从 MVP 进入 Phase 2），清理上一阶段的过时信息
- 模块完成时，清理该模块的临时状态

---

## 检查清单

每次更新 `active-context.md` 后确认：

- [ ] 只更新了属于自己角色的章节（或按冲突处理机制处理）
- [ ] 更新了 `## 最近变更` 记录本次变更
- [ ] 没有写入详细内容（只写状态和引用）
- [ ] 文件路径引用是正确的相对路径
- [ ] 状态标记使用统一符号（✅ 🔄 ⏳ ❌）
