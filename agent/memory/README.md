# Agent 记忆体系使用指南

## 双记忆体系架构

Strategy Compass 项目采用双记忆体系，分别服务于不同的场景和需求：

### 1. 项目级长期上下文（memory-bank/）

**位置**：`/agent/memory-bank/`

**定位**：项目的"大脑"，存储稳定的、长期有效的项目信息

**包含内容**：
- `projectbrief.md` - 项目概要、目标用户、核心价值
- `active-context.md` - 当前开发焦点、下一步行动
- `progress.md` - 整体进度追踪、里程碑状态
- `system-patterns.md` - 架构模式、模块边界、设计决策
- `tech-context.md` - 技术栈、外部API、已知限制
- `product-context.md` - 产品上下文、用户画像、竞品参考

**使用场景**：
- 新会话启动时加载，快速了解项目全貌
- 架构设计和技术决策时参考
- 产品规划和需求分析时使用
- 长期进度跟踪和里程碑管理

**更新规则**：
- 只在重大决策或里程碑变更时更新
- 保持信息的稳定性和权威性
- 由架构师或产品经理角色负责维护

### 2. 会话级工作日志（agent/memory/）

**位置**：`/agent/memory/`

**定位**：项目的"日记本"，存储临时的、会话相关的工作记录

**包含内容**：
- `YYYY-MM-DD.md` - 每日工作日志
- `session-framework.md` - 架构设计会话框架
- `*.md` - 特定主题的工作记录

**使用场景**：
- 记录每日开发工作和调试过程
- 存储临时决策和待验证的方案
- 会话间的上下文传递
- 问题排查和调试信息

**更新规则**：
- 每次会话结束时更新当日日志
- 记录具体的工作内容和遇到的问题
- 保留最近30天的日志，更早的日志归档
- 由具体执行任务的Agent负责记录

## 使用规范

### 会话启动时

1. **首先加载 memory-bank/**：了解项目全貌和当前状态
2. **然后检查 agent/memory/**：查看最近的工作日志，了解上下文
3. **确认当前任务**：根据 active-context.md 确定本次会话目标

### 工作过程中

1. **重要决策写入 memory-bank/**：架构变更、技术选型、产品决策等
2. **工作过程记录到 agent/memory/**：调试信息、临时方案、待验证假设等
3. **保持两套体系的独立性**：不要将会话级日志混入项目级上下文

### 会话结束时

1. **更新 agent/memory/YYYY-MM-DD.md**：记录本次会话的工作内容
2. **检查是否需要更新 memory-bank/**：如果有重大决策或里程碑变更
3. **清理临时文件**：删除不再需要的临时工作文件

## 常见问题

### Q: 什么时候更新 memory-bank/？

A: 只在以下情况更新：
- 完成重要里程碑（如模块设计完成、编码阶段完成）
- 做出重大技术决策（如技术栈变更、架构调整）
- 产品方向有重要变化（如目标用户调整、核心功能变更）

### Q: 什么时候更新 agent/memory/？

A: 以下情况都需要更新：
- 每次会话结束时
- 遇到重要问题或解决方案时
- 做出临时决策或待验证方案时
- 需要记录调试信息或排查过程时

### Q: 两套体系的信息冲突怎么办？

A: 以 memory-bank/ 为准。如果发现冲突，需要：
1. 检查 memory-bank/ 的信息是否过时
2. 如果过时，更新 memory-bank/ 并说明原因
3. 如果 agent/memory/ 的信息有误，更正日志记录

### Q: 如何避免信息重复？

A: 遵循以下原则：
- memory-bank/ 存储结论和决策
- agent/memory/ 存储过程和依据
- 定期清理过时的会话级日志

## 最佳实践

1. **保持 memory-bank/ 的简洁性**：只存储关键信息，避免冗余
2. **保持 agent/memory/ 的完整性**：详细记录工作过程，便于追溯
3. **定期整理和归档**：每月清理过时的会话级日志
4. **建立索引机制**：在 memory-bank/ 中索引重要的会话级记录
