# 架构设计会话框架

## 核心原则

> 架构不是画出来的，是一轮一轮决策出来的。

- 每次会话只做 **一个层级的一个决策**
- 不追求一次搞定全局，追求每次决策都可追溯
- 文件系统是持久记忆，对话是临时计算

## 设计划分

```
Layer 0 ─ 愿景层
  项目定位、目标用户、核心价值主张
  产出：SOUL.md、productContext.md

Layer 1 ─ 系统层
  整体技术栈、模块边界、数据流
  产出：systemPatterns.md、架构图

Layer 2 ─ 模块层（逐个模块）
  每个模块的接口设计、数据结构、核心逻辑
  产出：module-spec.md（per module）

Layer 3 ─ 实现层
  具体代码、配置、部署脚本
  产出：由编码 Agent 完成
```

**当前状态：** Layer 0 已完成。Layer 1 部分完成（技术选型已决策）。
**下一步：** 从 Layer 1 系统层的模块边界划分开始。

## 会话生命周期

```
┌─────────────────────────────────────────────┐
│  1. 启动（2-3 轮）                            │
│     用户："读取 memory-bank，开始设计 XX"      │
│     我：读取上下文 → 确认目标和范围            │
│                                              │
│  2. 探索（3-5 轮）                            │
│     调研方案 → 分析取舍 → 提出候选             │
│                                              │
│  3. 决策（2-3 轮）                            │
│     用户选定方案 → 我确认约束条件              │
│                                              │
│  4. 输出（2-3 轮）                            │
│     写入设计文档 → 更新 memory-bank/           │
│                                              │
│  5. 收尾（1 轮）                              │
│     确认交付物 → 标记进度 → 结束对话           │
│                                              │
│  总计：10-15 轮 / 会话                        │
└─────────────────────────────────────────────┘
```

## 会话启动模板

每次新对话时发送：

```
读取 memory-bank/activeContext.md 和 memory-bank/systemPatterns.md，
确认当前设计阶段和已有决策。
本会话目标：设计 [模块名] 的 [具体设计内容]。
```

## 会话结束检查清单

每次会话结束前，我必须完成：

- [ ] 设计决策已写入对应的文档（docs/ 或 memory-bank/）
- [ ] systemPatterns.md 已更新（如涉及架构级决策）
- [ ] progress.md 已更新（标记完成状态）
- [ ] activeContext.md 已更新（标记下一个待设计项）
- [ ] 如有阻塞项，已在 activeContext.md 中标记

## 分层决策的好处

1. **可追溯**：每个决策都有文件记录，"为什么这样设计"不会丢
2. **可调整**：修改上层决策时，下层受影响的模块可以逐个重新设计
3. **可并行**：不同模块的设计可以由不同对话（甚至不同 Agent）并行推进
4. **可验证**：每个层级的产出都有明确的验收标准
