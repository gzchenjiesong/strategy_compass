# 技术模式 (System Patterns)

## 模块架构（Layer 1 决策）

### 三层模块划分

```
┌─────────────────────────────────────────────────────────────┐
│                        应用层                                │

│      │                      │                   │            │
│      └──────────┬───────────┘     ┌─────────────┘            │
│                 │                 │                          │
│                 ▼                 ▼                          │
│  ┌────────────────────────────────────────────┐             │
│  │              分析层                          │             │
│  │   Module 4: 估值分析（百分位/估值区间）        │             │
│  └─────────────────────┬──────────────────────┘             │

│  ┌──────────────────────┼──────────────────────┐            │
│  │              数据层    ▼                       │            │

```

### 模块清单

| 模块 | 名称 | 层级 | 职责 |
|------|------|------|------|
| Module 0 | 用户系统 | 数据层（横跨） | 微信授权认证 + 邀请码 + 用户数据管理（自选/偏好/配置） |
| Module 1 | 数据底座 | 数据层 | 统一外部数据获取入口（行情/财务/基金/新闻原始数据） |
| Module 2 | 新闻聚合 | 数据层 | 新闻加工处理（分类/去重/标签/关联股票板块） |
| Module 3 | 市场概览 | 应用层 | 首页仪表盘（指数行情/市场温度/板块涨跌） |
| Module 4 | 估值分析 | 分析层 | 估值百分位计算 / 估值区间判断 / 历史估值曲线 |
| Module 5 | 板块概念 | 应用层 | 行业/概念板块趋势追踪与深度分析 |
| Module 6 | 个股基金详情 | 应用层 | 单个标的的深度信息展示（基本面/技术面/估值） |
| Module 7 | 通知服务 | 基础设施 | 微信推送 / 站内消息（MVP 后置） |

### 模块依赖关系

```
Module 0 (用户) ← 所有模块依赖（鉴权 + 用户数据隔离）
Module 1 (数据) ← Module 2, 3, 4, 5, 6 依赖（数据获取）
Module 4 (估值) ← Module 3, 6 依赖（估值分析能力）
Module 2 (新闻) ← 独立数据获取，通过标签关联其他模块
Module 3 (概览) ← 调用 Module 1 + Module 4
Module 5 (板块) ← 调用 Module 1
Module 6 (详情) ← 调用 Module 1 + Module 4
```

### 模块边界约束

1. **数据获取唯一入口**：所有外部数据通过 Module 1 获取，其他模块不直接调用外部 API
2. **估值能力统一出口**：所有估值计算通过 Module 4 提供，其他模块不重复实现
3. **用户数据隔离**：所有业务数据表必须包含 user_id 字段
4. **模块间通信**：通过 API 接口，不直接调用内部函数/类
5. **用户偏好影响展示**：Module 3/4/5/6 展示内容受 Module 0 中用户配置影响

### MVP 范围

**MVP 包含：** Module 0 + 1 + 2 + 3 + 4 + 5 + 6
**MVP 后置：** Module 7（通知）、Module 8（策略管理）

## 架构模式

### 整体架构
- **前后端分离** — 后端提供 RESTful API，前端独立构建部署
- **API 驱动** — 所有功能通过 API 暴露，Web/未来 App 共用同一套
- **单体应用（初期）** — 不拆微服务，单 Flask 应用 + SQLite 够用

### 后端分层
```
API Layer (路由 + 参数校验)
    ↓
Service Layer (业务逻辑)
    ↓
Data Layer (SQLite + 外部 API 调用)
```

### 数据库
- **初期 SQLite** — 简单、零运维、够用
- **WAL 模式** — 允许并发读，提升性能
- **volume 挂载** — Docker 容器中数据持久化
- **每日备份** — cron 脚本备份到 /tmp + 可选上传
- **用户隔离** — 所有业务表包含 user_id 字段

### 部署
- **Docker Compose** — Flask (Gunicorn) + Nginx
- **Nginx 反向代理** — 静态文件 + SSL + 限流
- **Let's Encrypt** — 免费 SSL 证书，自动续期

## 设计模式

### 数据获取
- **缓存优先** — 行情数据本地缓存，定时刷新
- **降级处理** — 外部 API 不可用时返回缓存数据
- **多源备用** — 关键数据有备用数据源
- **统一入口** — 所有外部数据通过 Module 1 获取

### 策略引擎（未来）
- **策略接口统一** — 每个策略实现统一的 `Strategy` 接口
- **配置驱动** — 策略参数通过数据库配置，不硬编码
- **事件驱动** — 定时任务扫描触发，非实时轮询

## 命名约定

- 数据库表：`snake_case`（如 `grid_configs`）
- Python 变量/函数：`snake_case`
- API 路径：`/api/v1/resource-name`
- 前端组件：`PascalCase`（如 `MarketCard.vue`）
