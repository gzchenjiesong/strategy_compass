# Skill 封装流程 (SOP)

> 当 Agent 被要求创建或封装一个 Skill 时，按此流程执行。

## 流程概览

```
需求判断 → 结构选择 → 编写内容 → 存放 → 验证
```

## 1. 需求判断：是否应该封装为 Skill

不是所有能力都需要封装为 Skill。判断标准：

| 应该封装为 Skill | 不需要，直接在会话中做 |
|------------------|----------------------|
| 需要被反复触发的能力 | 一次性任务 |
| 有明确触发关键词的能力 | 临时性的分析或讨论 |
| 涉及固定流程/模板的能力 | 简单的文件读写操作 |
| 跨会话复用的能力 | 当前会话内就能完成的工作 |

**自检问题：** "这个能力下次还会用到吗？触发条件明确吗？"
- 两个都是 → 封装为 Skill
- 有一个否 → 不封装

## 2. 结构选择：单文件 vs 目录

### 决策规则

```
这个 Skill 需要脚本、模板文件、配置等附属资源吗？
  ├── 否 → 单文件：agent/skills/{skill-name}-skill.md
  └── 是 → 目录：agent/skills/{skill-name}/
              ├── SKILL.md          （主文件，必须）
              ├── scripts/          （执行脚本，按需）
              ├── templates/        （模板文件，按需）
              └── config/           （配置文件，按需）
```

### 判断示例

| Skill 类型 | 结构 | 理由 |
|-----------|------|------|
| 提示词生成器 | 单文件 | 纯文档，无脚本 |
| 新闻聚合 | 目录 | 需要 Python 脚本采集数据 |
| 代码检查 | 目录 | 需要 lint 脚本 + 规则配置 |
| 文件格式转换 | 目录 | 需要转换脚本 |
| 会话模板 | 单文件 | 纯文档，模板内嵌在 SKILL.md 中 |
| 部署自动化 | 目录 | 需要部署脚本 + 配置模板 |

### 命名规范

- 单文件：`{skill-name}-skill.md`（带 `-skill` 后缀，与普通文件区分）
- 目录：`{skill-name}/`（不带后缀，`SKILL.md` 在目录内）
- 名称用小写 + 连字符：`startup-prompt-skill.md`、`news-aggregator/`

## 3. 存放路径

所有项目级 Skill 统一存放在：

```
agent/skills/
├── {skill-name}-skill.md      ← 单文件 Skill
├── {skill-name}/              ← 目录 Skill
│   ├── SKILL.md
│   └── scripts/
└── .gitkeep
```

**注意：** 这是项目级 Skill，与 OpenClaw 内置 Skill（`~/.openclaw/skills/`）区分开。

## 4. SKILL.md 内容规范

每个 Skill 的主文件（无论是单文件还是目录内的 SKILL.md）必须包含以下结构：

```markdown
---
name: {skill-name}
description: |
  简短描述，说明这个 Skill 做什么。Keywords: 触发词1, 触发词2, ...
---

# {Skill 名称}

## 概述
一段话说明这个 Skill 的用途和适用场景。

## 触发关键词
列出用户说了什么话时会触发这个 Skill。

## 前置依赖（如有）
需要安装的工具、库、API Key 等。

## 使用流程
描述 Agent 执行这个 Skill 的完整流程，用步骤 + checklist 格式。

## 输出规范
说明 Skill 执行后产出什么、存放在哪里。

## 定制指南（如有）
根据不同场景如何调整行为。
```

### 必须包含的部分
- ✅ YAML frontmatter（name + description）
- ✅ 概述
- ✅ 触发关键词
- ✅ 使用流程

### 按需包含的部分
- ⚪ 前置依赖
- ⚪ 命令列表（脚本型 Skill）
- ⚪ 输出规范
- ⚪ 定制指南
- ⚪ 使用示例

## 5. 编写流程

### Step 1: 确定 Skill 范围

- [ ] 明确 Skill 做什么、不做什么
- [ ] 确定触发关键词（至少 3 个，中英文各备）
- [ ] 判断是否需要脚本 → 决定单文件或目录

### Step 2: 编写 SKILL.md

- [ ] 按内容规范编写
- [ ] 流程步骤用 checklist 格式（`- [ ]`）
- [ ] 如有脚本，说明调用方式和参数

### Step 3: 编写附属文件（目录型）

- [ ] `scripts/` — 执行脚本，添加执行权限（`chmod +x`）
- [ ] `templates/` — 模板文件，使用清晰的占位符格式
- [ ] `config/` — 配置文件，提供默认值

### Step 4: 存放

- [ ] 单文件 → `agent/skills/{name}-skill.md`
- [ ] 目录 → `agent/skills/{name}/SKILL.md` + 附属文件
- [ ] 确认文件编码为 UTF-8

### Step 5: 验证

- [ ] 在会话中用触发词测试，确认 Agent 能正确识别并执行
- [ ] 检查输出是否符合预期格式
- [ ] 如有问题，迭代 SKILL.md 中的流程描述

## 6. 检查清单

每次封装 Skill 后过一遍：

- [ ] 文件名/目录名符合命名规范？
- [ ] SKILL.md 包含必须的部分（frontmatter + 概述 + 触发词 + 流程）？
- [ ] 触发关键词覆盖中英文常见说法？
- [ ] 流程步骤清晰、可执行？
- [ ] 存放在 `agent/skills/` 下？
- [ ] 单文件/目录选择合理？
- [ ] 测试过触发效果？
