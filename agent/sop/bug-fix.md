# Bug 修复流程 (SOP)

> 当 Agent 被要求修复一个 Bug 时，按此流程执行。

## 流程概览

```
复现 → 定位 → 修复 → 验证 → 记录 → 扫雷
```

## 步骤详解

### 1. 复现 Bug

- [ ] 确认 Bug 的具体表现（错误信息、截图、日志）
- [ ] 在本地环境复现（`docker-compose up` 或 `python wsgi.py`）
- [ ] 如果无法复现，记录环境差异，先不修
- [ ] 记录复现步骤（便于验证修复）

### 2. 定位原因

- [ ] 先读错误信息和日志（`backend/logs/` 或容器日志）
- [ ] 读相关代码，理解上下文
- [ ] 形成假设 → 验证假设 → 确认根因
- [ ] **不要盲目改代码，先理解**
- [ ] 如果涉及数据问题，检查 SQLite 中的相关数据

### 3. 修复

- [ ] 最小改动原则：只改必须改的部分
- [ ] 不要在修 Bug 时顺手重构（分开做）
- [ ] 考虑修复是否会影响其他功能
- [ ] 检查同类场景：**"还有没有类似的？"**

> **扫雷原则**：修复一个 Bug 时，主动扫描同类场景。例如修复了"游戏 ETF 显示错误"，要检查其他 ETF 是否也有类似问题。

### 4. 验证

- [ ] 本地复现 → 修复 → 确认不再复现
- [ ] 运行测试：`pytest tests/`
- [ ] 如果 Bug 涉及 API，添加对应的测试用例
- [ ] 检查是否引入了新的问题（回归测试）

### 5. 记录

- [ ] Git commit message：`fix: 简要描述`
- [ ] 如果是重大 Bug，在 `agent/memory/YYYY-MM-DD.md` 记录根因和教训
- [ ] 如果发现系统性问题，在 `agent/skills/` 沉淀经验

### 6. 扫雷（关键步骤）

修复完成后必须执行：

```
修复了 X 问题
→ 搜索代码中所有类似 X 的模式
→ 检查是否有相同根因的隐藏 Bug
→ 如有，一并修复
```

示例：
- 修复了"数据源映射错误导致显示错误名称"
- → 检查所有数据源映射点
- → 发现还有 3 个地方用了相同的错误映射逻辑
- → 一并修复

## 常见陷阱

| 陷阱 | 正确做法 |
|------|----------|
| 改了表面症状没修根因 | 找到 WHY，不只是修 WHAT |
| 修 Bug 时顺手重构 | 分开做，先修 Bug 再重构 |
| 没测试就提交 | 本地验证 + pytest 必须通过 |
| 改了很多文件 | 最小改动，diff 应该很小 |
| 修完不扫雷 | 主动扫描同类场景，防止遗漏 |
| 不记录根因 | 重大 Bug 必须记录到 memory |
